Why Claude/Cursor over Gemini CLI?
Elektra likely has Gemini via GSuite ‚Äî but Gemini CLI's coding agent lags meaningfully behind Claude Code and Cursor today in code quality, context handling, and agentic reliability. For a time-boxed pilot where output quality matters, Claude Code + Cursor is the stronger choice. That said, the pipeline should be built model-agnostic ‚Äî swap the model string in one config file and the whole system works with Gemini, GPT-4, or whatever outperforms in 6 months. We're not betting on a model; we're betting on the workflow.

Approximate Build ‚Äî What Gets Created in the Pilot
Two AI Prompt Pipelines
A script that takes a clinical note as input, sends it to the Claude API with a structured prompt, and returns suggested ICD-10 + CPT codes with rationale. A second script takes a denial reason code + clinical note and returns a draft appeal letter in Elektra's voice and format.
A Simple Review UI
A lightweight web interface (like the sample dashboard below) where the billing team can see AI-generated suggestions, edit them, approve or reject, and track status. No EHR integration required in the pilot ‚Äî manual data entry to test the concept. Built with Claude Code in ~1 week.
Built Model-Agnostic from Day One
The pipeline is architected so the AI model is a swappable config value ‚Äî not hardcoded. If Gemini 3 or GPT-5 outperforms Claude on billing tasks in 6 months, we point the config at the new model and the whole system follows. We're building a durable workflow, not a dependency on any single vendor's roadmap.

04 Build vs. Buy vs. Neither ‚Äî Decision Framework
‚ö† The Vendor Trap
Don't pay $50k/year for a vendor wrapper to answer a question a 6-week pilot can answer for a few thousand dollars in engineering time.
Most healthcare billing AI vendors are not building proprietary models ‚Äî they're wrapping the same general-purpose APIs (Claude, GPT-4) in a healthcare-branded UI, adding billing-specific prompts, and charging enterprise prices for it. The core intelligence is not theirs. The pilot is specifically designed to validate ROI before you're locked into a contract, so that if you do eventually buy, you know exactly what you're paying for and have the leverage to negotiate it.
üî®
Build
Choose Build When:
  • Pilot proves clear time savings (2+ hrs/week recovered)
  • No vendor fits Elektra's specific payer mix or women's health workflows
  • You want full control over the prompt logic and AI behavior
  • Volume is growing and the ROI compounds over time
  • You have engineering capacity (even part-time)
üõí
Buy
Choose Buy When:
  • Pilot confirms ROI and a vendor solution exists at the right price
  • Engineering capacity is constrained or costly
  • Vendor offers EHR integration that would be expensive to build
  • Compliance and audit requirements make a certified vendor preferable
  • The problem is table stakes, not a competitive differentiator
‚è∏
Neither (Yet)
Pause When:
  • Pilot shows current volume is too low to justify investment
  • Manual process is already fast and accurate enough
  • Implementation overhead exceeds expected savings in year 1
  • Team doesn't have bandwidth to manage an AI tool or vendor
  • Data quality is too poor to produce reliable AI outputs
